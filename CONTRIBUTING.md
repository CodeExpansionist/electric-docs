# Contributing to Electric

## Prerequisites

- Node.js 18+
- npm 9+

## Setup

```bash
git clone <repo-url>
cd electriz
npm install
bash scripts/install-git-hooks.sh  # Install pre-commit drift checks
npm run dev
```

Open [http://localhost:3001](http://localhost:3001).

## Code Standards

### Styling

- Use Tailwind CSS utility classes with custom design tokens from `data/design-tokens.json`.
- Token values take priority over Tailwind defaults. Use `bg-primary`, `text-text-primary`, `bg-surface` etc.
- See [docs/DESIGN_SYSTEM.md](docs/DESIGN_SYSTEM.md) for the full token reference.

### No External Dependencies at Runtime

This is a fully self-contained MVP. Every image is local, every data file is static JSON. Do not:

- Add external image URLs to `<img>` or `next/image` `src` attributes
- Add API calls to external services
- Add `remotePatterns` entries to `next.config.mjs`

### Data

All content comes from `data/scrape/` JSON files. No hardcoded product names, prices, or descriptions in components. See [docs/DATA_SCHEMA.md](docs/DATA_SCHEMA.md) for the data model.

### Images

Product images live at `public/images/products/{productId}/{type}.webp`. Use the helpers in `src/lib/images.ts` (`getListingImage`, `getLargeImage`, `getGalleryImages`) instead of constructing paths manually.

### TypeScript

- Strict mode enabled
- Prefer explicit types over `any` (use `eslint-disable` sparingly for scraped data parsing)

### Category Routing

URL slugs must match keys in `categoryMap` in `src/lib/category-data.ts`. See [docs/CATEGORY_ROUTING.md](docs/CATEGORY_ROUTING.md) for the full routing algorithm.

## Commit Conventions

```
feat(section): add [page] [section-name]
fix(component): resolve [issue description]
chore: [maintenance task]
```

Do not commit:
- `.env` files (use `.env.example` as template)
- The `.next/` build cache
- `node_modules/`

## Dev Server Gotcha

Never delete `.next/` while the dev server is running. Kill the server first:

```bash
lsof -ti:3001 | xargs kill -9
rm -rf .next
npm run dev
```

## Documentation Drift

If your changes touch `.md` files, `package.json`, `scripts/`, `.env.example`, or `data/repo-facts.json`, run:

```bash
npm run preflight
```

This regenerates repo facts and audits docs for drift. The pre-commit hook runs this automatically for staged drift-relevant files. See [docs/GIT_HOOKS.md](docs/GIT_HOOKS.md).

## Documentation Standards

### Authority Model

Every document in `docs/` falls into one of three tiers:

| Tier | Rule | Example |
| --- | --- | --- |
| **Governance** | Doc defines intended behaviour. If code deviates, fix the code (or update the doc if policy is changing). | `CLAUDE.md`, `DEPLOYMENT.md`, `KNOWN_ISSUES.md` |
| **Reference** | Doc describes code. If doc and code disagree, update the doc. | `DATA_SCHEMA.md`, `COMPONENT_GUIDE.md`, `CHECKOUT_FLOW.md` |
| **Generated** | Tool output. Never hand-edit. Fix the tool or data, then regenerate. | `data/repo-facts.json`, `parity-report.md` |

Each doc states its authority in a blockquote immediately after the title.

### When to Update Docs

Any change set that alters behaviour, routing, architecture, APIs, deployment, or workflow must include the corresponding documentation update in the same commit unless explicitly marked as a deferred docs follow-up (e.g., `docs-follow-up: COMPONENT_GUIDE.md` in the commit message).

| Change Type | Document to Update |
| --- | --- |
| New component | `docs/COMPONENT_GUIDE.md` |
| New API route | `docs/API_REFERENCE.md` |
| Routing logic | `docs/CATEGORY_ROUTING.md` |
| Pipeline script | `docs/DATA_PIPELINE.md` |
| Design token | `docs/DESIGN_SYSTEM.md` |
| Bug found/fixed | `docs/KNOWN_ISSUES.md` |
| Architecture decision | `docs/ARCHITECTURE.md` |
| Deployment config | `docs/DEPLOYMENT.md` |

### Generated Artifacts

Two artifacts are generated by tooling, not maintained by hand:

| Artifact | Generated By | Purpose |
| --- | --- | --- |
| `data/repo-facts.json` | `npm run facts` | Product count, image count, component count, route count. Docs should reference these numbers, not hardcode them. |
| `artifacts/parity/reports/parity-report.md` | `npm run test:parity` | Automated parity test results. `KNOWN_ISSUES.md` cross-references the report date and verdict but does not duplicate its content. |

### Freshness Rules

Docs that describe current state must show their currency:

- `KNOWN_ISSUES.md` must always display a "Last audited" date
- Any doc stating current counts should cite `data/repo-facts.json` or include a "Last updated" marker
- If a status doc has no date marker, treat its contents as unverified

## Pull Request Checklist

- [ ] Changes match the reference screenshots in `reference-screenshots/`
- [ ] No external image URLs or API calls added
- [ ] Design tokens used instead of hardcoded colors/sizes
- [ ] `npm run build` passes without errors
- [ ] `npm run lint` passes
- [ ] `npm run preflight` passes (if docs, scripts, or package.json changed)
- [ ] Documentation updated for any behaviour, API, or architecture changes
