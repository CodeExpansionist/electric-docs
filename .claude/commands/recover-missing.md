Recover products that failed normal scraping using a three-tier strategy — retry with locale fix, targeted extraction, and autonomous agent as last resort.

**Pipeline position:** 5/32 — depends on: `/scrape-products` | feeds into: `/normalize-data`, `/download-images`

**Usage:** `/recover-missing` — processes `data/scrape/missing-products.json`

Optionally: `/recover-missing <category>` where `$ARGUMENTS` is a category to filter (e.g., `televisions`). Omit to process all missing products.

---

## Golden Rule

**Real data only.** Every recovered product must have data scraped from the actual site. The three-tier strategy exists to handle different failure modes — not to generate or approximate data. If all three tiers fail for a product, it stays in the missing list.

---

## Prerequisites

- `data/scrape/missing-products.json` must exist (generated by `/scrape-products` or `/audit-data`)
- `data/scrape/products/` directory must exist
- Firecrawl MCP tools must be available
- Reference site must be accessible

---

## Firecrawl Settings

**Locale:** ALL `firecrawl_scrape`, `firecrawl_extract`, and `firecrawl_agent` calls MUST include:
`location: { country: "{country}", languages: ["{language}"] }`
Many missing products failed precisely because the correct locale was absent — default US proxies may return 404 or redirect for locale-specific product pages.

**Cache Strategy:** Always use `maxAge: 0` — missing products failed previously, so cached results would just return the same failure.

## Steps

### 1. Load and categorize missing products

Read `data/scrape/missing-products.json`. Group products by category. If `$ARGUMENTS` is provided, filter to that category only.

Log the scope:

```
Missing products: X total
- {Category 1}: X
- {Category 2}: X
- ...
```

### 2. Attempt A — Retry with correct locale + scroll actions (cheapest)

For each missing product, retry with the fixes most likely to resolve the original failure:

```
firecrawl_scrape with:
  url: [product-url]
  formats: ["json"]
  location: { country: "{country}", languages: ["{language}"] }
  maxAge: 0
  actions: [
    { type: "wait", milliseconds: 2000 },
    { type: "scroll", direction: "down" },
    { type: "wait", milliseconds: 1000 }
  ]
  jsonOptions: {
    prompt: "Extract the full product detail including name, brand, price, rating, all specifications grouped by section, description, gallery image URLs, thumbnail URLs, key specs, delivery info, size/color variants, cross-sell products, care plans, and offers",
    schema: { ... same schema as /scrape-products step 4 ... }
  }
```

**Cost:** ~1 credit per product. Process in batches of 20 parallel calls.

**Success criteria:** Product has a non-empty `name`, non-zero `price.current`, and at least 1 gallery image URL.

Log results: "Attempt A: X/Y recovered, Z still failing"

### 3. Attempt B — Targeted extraction for sparse results

For products where Attempt A returned data but it was sparse (has name/price but empty `specifications` or no `description`), use `firecrawl_extract` with a focused schema targeting just the missing fields:

```
firecrawl_extract with:
  urls: [product-urls that returned sparse data]
  location: { country: "{country}", languages: ["{language}"] }
  prompt: "Extract all product specifications grouped by section, the full product description, and all gallery image URLs"
  schema: {
    type: "object",
    properties: {
      specifications: { type: "object" },
      description: {
        type: "object",
        properties: {
          main: { type: "string" },
          features: { type: "array", items: { type: "string" } },
          goodToKnow: { type: "array", items: { type: "string" } }
        }
      },
      galleryImages: { type: "array", items: { type: "string" } }
    }
  }
```

Merge the extracted fields into the partial data from Attempt A.

**Cost:** Moderate (LLM extraction). Process in batches of 20 (max 20 parallel scrapes per CLAUDE.md guidance).

Log results: "Attempt B: X/Y enriched, Z still incomplete"

### 4. Attempt C — Autonomous agent (last resort, expensive)

For products where both A and B failed completely (no data returned at all), use `firecrawl_agent`:

```
firecrawl_agent with:
  urls: [product-url]
  prompt: "Navigate to this product page and extract: product name, brand, current price, was price if shown, all specifications grouped by section, product description, all gallery image URLs, key specs with icons, delivery info, size and color variants, cross-sell products, care and repair plans, and any promotional offers. If the page redirects, follow the redirect. If a cookie consent dialog appears, dismiss it."
  schema: { ... full product detail schema ... }
```

Poll `firecrawl_agent_status` every 15 seconds, for up to 3 minutes per product.

**Cost:** Dynamic and expensive. Limit to genuine hard cases — do NOT run this on more than 50 products without user confirmation.

**Important:** If more than 20 products need Attempt C, pause and ask the user whether to proceed. Report the estimated credit cost.

Log results: "Attempt C: X/Y recovered via agent, Z permanently failed"

### 5. Save recovered products

For each successfully recovered product, write to `data/scrape/products/{productId}.json` using the same schema as `/scrape-products`.

### 6. Update missing products list

Update `data/scrape/missing-products.json`:
- Remove successfully recovered products
- For still-missing products, update with `failureReason` and `lastAttempt` timestamp:

```json
{
  "productId": "10XXXXXX",
  "url": "https://...",
  "category": "{Category}",
  "failureReason": "All 3 tiers failed — page returns 404 even with correct locale",
  "lastAttempt": "2026-03-06T12:00:00Z",
  "attemptsHistory": ["A: 404", "B: no data", "C: agent timeout"]
}
```

### 7. Output recovery report


## Critical Rules

- **Cheapest tier first.** Always run Attempt A before B, and B before C. Most failures are locale-related and resolve at tier A for 1 credit each.
- **Never fabricate data.** If all three tiers fail, the product stays missing. A missing product is better than fake data.
- **Respect credit budgets.** Attempt C (agent) is expensive. Never run it on more than 50 products without user confirmation.
- **Batch appropriately.** A: batches of 20. B: batches of 20. C: one at a time (expensive, needs monitoring). Max 20 parallel scrapes per batch across all tiers.
- **Merge, don't overwrite.** If Attempt A returns partial data and B enriches it, merge the fields. Don't discard A's price just because B extracted better specs.
- **Gallery image order is unreliable.** Firecrawl returns gallery URLs in DOM order, not display order. ~5% of products will have the base image (no `_NNN` suffix) buried mid-array. Don't assume gallery[0] is the hero image in raw scraped data — `mergeScrapedData()` sorts by suffix at runtime via `getImageSortKey()`. Run `/normalize-data` after recovery to fix ordering in source files too.
- **Log everything.** Every attempt's outcome goes in `attemptsHistory` so future runs don't re-attempt products with known permanent failures (e.g., 404s).
