Bootstrap a complete Next.js project scaffold from the site map output. Generates all route files, type definitions, and data layer — so building starts from a working skeleton, not a blank slate.

**Pipeline position:** 12/32 — depends on: `/map-site`, `/normalize-data`, `/verify-coverage` | feeds into: `/build-component`, `/build-admin`

**Usage:** `/scaffold-project` — generates scaffold from the category index (e.g., `categoryMap` or equivalent), `src/app/` route structure, and normalized data files

---

## Golden Rule

**Never guess — generate from real data.** Every route, type, slug, and import is derived from actual scraped data. No placeholder pages. No assumed URL patterns. No invented TypeScript types. If the site map says the slug is `{parent-category}/{child-category}/{grandchild-category}`, that's what the route handles — not a simplified version.

---

## Prerequisites

- The category data module must exist with a category index (e.g., `categoryMap`) defining all category slugs — path from `project-config.md`
- `data/scrape/routing-patterns.md` should exist (generated by `/map-site` Step 3b)
- At least one normalized category JSON in `data/scrape/` (run `/normalize-data` first)
- At least one product detail JSON in `data/scrape/products/`

---

## Steps

### 1. Analyze route structure from site map

Read the category index keys (e.g., `categoryMap`) from the category data module and the `src/app/` directory structure to determine:

1. **Route depth:** Count path segments per category URL. If ANY category has 2+ segments, use `[...category]` catch-all route (not `[category]` single-segment)
2. **Product URL pattern:** Extract how product URLs are structured (e.g., `/products/{slug}` vs `/products/{id}`)
3. **Hub pages vs listing pages:** Identify which URLs are hub/landing pages (editorial content) vs category listings (product grids)
4. **Static pages:** Identify non-category pages (basket, checkout, search, admin)

**Output decision log:**
```
Route analysis:
- Category route type: [...category] catch-all (3-segment slugs found)
- Product route type: [slug] (slug = slugified product name)
- Hub pages: {N} (section landing, sub-category hubs as discovered in site map)
- Static pages: {N} (basket, checkout, checkout/payment, search, admin)
```

### 2. Generate TypeScript types from actual JSON schema

Read 3-5 product detail JSON files from `data/scrape/products/` and generate `src/lib/types.ts`:

1. Analyze actual field names and value types across the sample
2. Generate interfaces that match the REAL data structure — not an assumed one
3. Include all fields found, mark optional fields with `?` based on which fields are missing in some products

**Key types to generate:**

- `Product` — the listing-level product (from category JSON)
- `ProductDetail` — the full product detail (from individual product JSON)
- `CategoryData` — the category listing with products array
- `SizeVariant`, `ColorVariant` — variant types
- `Price` — always `{ current: number; was: number | null; savings: number | null }`
- `Rating` — always `{ average: number; count: number }`
- `Specification`, `Description`, `DeliveryInfo`, etc.

**Critical:** Read the actual JSON first. Do NOT copy types from a previous project or assume field names. The `/normalize-data` step standardizes names, but verify the actual output.

### 3. Generate category data module

Create `src/lib/category-data.ts` with:

1. **Category index** (e.g., `categoryMap`) — keyed by exact URL slugs. Every key must match a real slug. **Values are factory functions `() => CategoryData`**, not plain objects — this enables lazy loading of category data.
2. **Import statements** — one per category JSON file. Use the tsconfig alias: `import categoryData from '@/../data/scrape/{category-filename}.json'`
3. **Banner configs** — per-category banner text, extracted from hub page data if available
4. **Filter definitions** — brand list, price ranges, dimension/size ranges per category (extracted from actual product data)
5. **Additional exports required:**
   - `getCategoryData(slug)` — resolves slug (including aliases) and invokes the factory function
   - `isHubCategory(slug)` — returns whether a category is a hub/landing page
   - `getCategoryHubData(slug)` — returns hub page editorial data for hub categories
   - `categoryAliases` — maps alternate slugs to canonical category index keys
   - `hubMap` — maps hub category slugs to their hub data sources

**Slug-to-filename mapping:** Category file names do NOT mechanically derive from URL slugs. Read the actual files in `data/scrape/` and map each to its corresponding slug based on the category name/URL inside the file.

**Static imports only:**
```typescript
// CORRECT — static JSON import with tsconfig alias
import categoryData from '@/../data/scrape/{category-filename}.json';

// WRONG — relative path traversal (fragile, hard to refactor)
import categoryData from '../../../data/scrape/{category-filename}.json';

// WRONG — never use these in Next.js App Router
import.meta.glob(...)
fs.readFileSync(...)
require(...)
```

### 4. Generate route files

Create all page files in `src/app/`:

**Category listing page:** `src/app/{section-slug}/[...category]/page.tsx`
- Client component (`'use client'`)
- Reads slug from params, looks up in the category index
- Renders product grid with filters
- Include fuzzy matching fallback for slug resolution (but log warnings when it activates)

**Product detail page:** `src/app/products/[slug]/page.tsx`
- Server component where possible
- Loads product from `products-index.json` by slug lookup

**Hub/landing pages:** One per hub URL discovered
- `src/app/{section-slug}/page.tsx` — section landing

**Static pages:**
- `src/app/basket/page.tsx`
- `src/app/checkout/page.tsx` + `src/app/checkout/payment/page.tsx`
- `src/app/search/page.tsx`
- `src/app/admin/page.tsx`
- `src/app/sitemap.ts`

Each page file is a working skeleton — renders data, handles loading/error states, ready for component building.

**Cart/basket architecture requirements (for any e-commerce clone):**

Before scaffolding the cart page, extract the reference's cart layout structure:

1. **Column count** for product item cards (2-col or 3-col — inspect, don't assume)
2. **Element flow** within the info area (title → qty/actions → price → badges → delivery)
3. **Sidebar content** (payment options, line items, promo, totals)

Architecture patterns:

- Cart state: Context + reducer pattern with localStorage persistence
- Default demo items: populate ALL product fields (including optional fields like energyRating, price.was) so the cart page renders realistically during development
- Service state (add-ons like warranties, installation): lift to page level, pass via props — don't isolate in child component state
- Summary component: accept computed values (service count, savings, discounts) as props, don't re-derive

**Rule: Cart item cards = listing cards in displayed metadata.** If the listing page card component renders a field, the cart item card must render it too. Compare the two components' props/fields before shipping.

### 5. Generate data loading utilities

Create `src/lib/product-data.ts`:

- `getProductBySlug(slug)` — looks up product in index by URL slug
- `getAllProducts()` — returns full product list from `products-index.json`
- `getSizeVariants(productId)` — returns size variant group for a product (e.g., same model in different sizes)

Create `src/lib/search-data.ts`:

- `searchProducts(query)` — basic text search across product names/brands

Create `src/lib/images.ts` with all 6 exported functions:

- `toLocalImage(cdnUrl)` — CDN URL to local path mapping (handles standard, variant, and M-prefix patterns)
- `toLocalImages(cdnUrls)` — batch version of `toLocalImage()` for arrays
- `getListingImage(productId)` — listing thumbnail path (`/images/products/{id}/main.webp`)
- `getLargeImage(productId)` — gallery main image path (`/images/products/{id}/large.webp`)
- `getGalleryImages(productId, count)` — array of gallery image paths
- `getThumbnailImages(productId, count)` — array of thumbnail image paths
- Use regex patterns from `/download-images` CDN pattern discovery

### 6. Generate layout components (shells only)

Create shell components in `src/components/layout/`:
- `Header.tsx` — with navigation data from site map
- `Footer.tsx` — with footer columns from site map
- `MainNav.tsx` — with category links from the category index
- `LayoutWrapper.tsx` — wraps pages with header/footer

Each component is a working shell that renders real data from the site map. Styling comes later via `/build-component`.

### 7. Generate context providers

Create React Context files:
- `src/lib/cart-context.tsx` — cart/basket state + localStorage persistence
- `src/lib/saved-context.tsx` — saved items state
- `src/lib/orders-context.tsx` — order history state (with seed data using REAL product IDs from scraped data, NOT invented IDs)

### 8. Generate config files

- `next.config.mjs` — with empty `remotePatterns` (all images local)
- `tailwind.config.ts` — with design tokens from `data/design-tokens.json`
- `tsconfig.json` — with path aliases (`@data`, `@components`, `@lib`)

### 9. Validate scaffold

After generating all files:

1. Run `npx tsc --noEmit` — verify zero type errors
2. Verify every category index key matches a slug cross-checked against the category data module
3. Verify every static import resolves to an existing file
4. Verify seed data product IDs exist in `data/scrape/products/`
5. Run `npm run dev` — verify the app starts without crashes

## Critical Rules

- **Generate from data, never from memory.** Read the category index from the category data module and actual JSON files. Never copy types or routes from a previous project — every site has different structures.
- **Static imports only.** Next.js App Router does not support `import.meta.glob` or `fs.readFileSync` in client components. Use static JSON imports with TypeScript path aliases.
- **Use `[...category]` catch-all for multi-segment slugs.** If ANY category has 2+ path segments (e.g., `{parent-category}/{child-category}`), the single `[category]` route won't work.
- **Real product IDs in seed data.** Context providers with demo/seed data must reference product IDs that actually exist in `data/scrape/products/`. Never invent product IDs.
- **`toLocalImage()` for every image.** Every `<Image>` component must route through `toLocalImage()`. No direct CDN URLs.
- **`product.productId` as React key.** Never use array index as key for product lists. Always use the unique product ID.
